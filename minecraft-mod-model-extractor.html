DOCUMENTATION

How the extractor works

A technical walkthrough of the pipeline, the file formats involved, and what to expect from the output.

1. What counts as a valid input

Upload any Minecraft mod packaged as a .jar file. A .jar is just a ZIP archive under the hood, so the extractor opens it the same way a ZIP tool would — no Minecraft installation or Java runtime is needed.

2. Recursive .tbl discovery

Once the archive is open, every entry in it — regardless of how deeply nested its folder is — is checked against the .tbl extension. This means models buried in paths like assets/mymod/models/entity/sub/part.tbl are found exactly the same as ones sitting at the root.

3. What a .tbl file actually contains

A Tabula .tbl file is itself a small ZIP archive containing:

  • model.json — the cuboid hierarchy: every box's position, rotation, dimensions, and texture offset
  • a texture image (usually .png) — the skin applied to the model

The extractor opens this inner archive for every discovered .tbl, independently of the others.

4. The conversion itself

Each cuboid becomes a glTF node with its own mesh: 24 vertices (4 per face), correct face normals, and UV coordinates computed from the same cross-unwrap layout Tabula/Techne and Minecraft itself use. Parent/child relationships in the cuboid tree become the glTF node hierarchy, so pivots and rotations stay intact.

model.json (cuboid tree)
  └─ BodyUpper
       ├─ BodyLower
       ├─ Wing_L
       └─ Wing_R
            └─ WingTip_R

→ becomes a glTF node tree with the same structure,
  each node carrying its own translation / rotation / mesh.

5. Output structure

Every successfully converted model is written into an output ZIP at the same relative path it had inside the original mod, with only the extension changed:

assets/dragonmod/models/dragon.tbl   →  assets/dragonmod/models/dragon.gltf
assets/dragonmod/models/sub/wings.tbl →  assets/dragonmod/models/sub/wings.gltf

This means the folder hierarchy of the original mod is preserved exactly in the output ZIP.

6. Error handling

If a particular .tbl is corrupted, empty, or not actually a valid archive, that one file is marked as failed and skipped — the rest of the mod continues converting normally. You'll see exactly which files failed, and why, in the results list.

7. Performance & memory

Models are converted one at a time rather than all at once, so memory usage stays flat even on mods containing hundreds of models — nothing is held in memory longer than the single file currently being processed.

8. Privacy

Everything — extraction, conversion, and packaging — happens locally in your browser tab. Your mod file is never uploaded anywhere. See the Privacy Policy for details.

FAQ

Frequently asked questions

Common questions about the extractor, the formats it supports, and its limits.

Does my mod file get uploaded anywhere?▾
No. Everything runs client-side in your browser — extraction, model conversion, and ZIP packaging never leave your device. There is no server component.
What model formats are supported as input?▾
Only Tabula .tbl files, found anywhere inside the uploaded .jar. Other model formats (Bedrock geometry JSON, OptiFine JPM/JEM, Blockbench-native projects that aren't .tbl-based) aren't scanned for — only genuine Tabula/Techne-style .tbl archives are recognized.
Why did some models fail to convert?▾
Usually because the .tbl file is corrupted, empty, or doesn't actually contain a model.json inside it. The results list shows exactly which files failed and why — every other model in the mod still converts normally.
Will textures be included in the output?▾
Yes, when the original .tbl includes one. The texture is embedded directly inside each .gltf file as base64 data, so every converted model is a single self-contained file — no separate texture files to manage.
Does the output preserve the mod's folder structure?▾
Yes. Each converted model is written into the output ZIP at the same relative path the original .tbl had — only the file extension changes.
How large a mod can this handle?▾
Models are converted one at a time rather than all simultaneously, which keeps memory usage flat regardless of how many models a mod contains. Extremely large mods (hundreds of MB, thousands of models) will simply take longer, limited mainly by your device and browser.
Can I use the converted .gltf files commercially?▾
The conversion itself doesn't grant you any rights to a mod's assets — that depends entirely on the original mod's license. This tool only changes the file format; it doesn't change who owns the content.
Why glTF specifically?▾
glTF 2.0 is the most widely supported modern 3D interchange format — it opens directly in Blender, three.js, Babylon.js, most game engines, and countless online viewers, without needing any Minecraft-specific tooling.

PRIVACY POLICY

Your files never leave your browser

Last updated 2026. This tool was built to be private by construction, not by policy.

No uploads, no server

This application has no backend. There is no server that receives, stores, or processes your .jar file. Every step — reading the archive, scanning for models, converting them, and building the output ZIP — runs entirely inside your own browser tab using JavaScript.

What we don't collect

  • We don't collect, log, or transmit the contents of any file you upload.
  • We don't store filenames, model data, or textures anywhere.
  • We don't use analytics or tracking scripts on this site.
  • We don't use cookies for tracking purposes.

Third-party resources

The page loads two external resources over HTTPS: Google Fonts (for typography) and the JSZip library from a public CDN (for reading and writing ZIP/JAR archives). Neither receives your uploaded files — they are static assets loaded once when the page opens.
